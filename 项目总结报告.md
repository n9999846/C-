# 酒店客房管理系统 — 项目总结报告

> 第 17-18 周 | 项目周期：2026年6月

---

## 一、项目介绍

### 1.1 项目背景

随着酒店行业信息化管理水平的不断提升，传统的人工记账与纸质登记方式已难以满足现代酒店对客房管理、客户维护和营收分析的需求。本项目以 C++ 面向对象程序设计课程的大作业为载体，设计并实现了一套功能完整的**酒店客房管理系统**，旨在综合运用课堂所学的 OOP 核心知识，同时通过实际项目锻炼团队协作与软件工程能力。

### 1.2 项目需求

系统需覆盖酒店日常运营的核心业务流程，具体包括：

| 功能模块 | 需求描述 |
|---------|---------|
| **房间管理** | 支持标准间、豪华间、套房三种房型，涵盖房间的增删改查、状态管理、多条件筛选与排序 |
| **客户管理** | 支持普通客户与 VIP 客户的注册、信息维护、VIP 升级，VIP 客户享有不同等级折扣 |
| **预订管理** | 完整的预订生命周期：创建预订 → 入住 → 退房/取消/续住，含日期冲突检测与异常处理 |
| **账单计费** | 退房时自动生成账单，计算房费、服务费、折扣，支持支付处理 |
| **营收统计** | 日/月营收统计、按房型统计、消费排行 Top N 客户、月度趋势分析 |
| **数据持久化** | 所有数据通过文件存储（rooms.txt、customers.txt、bookings.txt、bills.txt、system.log），支持启动加载、操作实时保存与关闭保存 |
| **图形界面** | 基于 Qt 框架实现桌面 GUI，包含登录、仪表盘、各业务管理页面 |

### 1.3 项目作用

- **教学层面**：全面覆盖 C++ OOP 核心知识点——继承（单继承、二次继承）、多态（虚函数、抽象类）、运算符重载、友元、异常处理、STL（vector、map、sort、accumulate、count_if、lambda）、单例模式、门面模式、静态工具类等
- **实践层面**：体验从需求分析、模块设计、编码实现到集成测试的完整软件开发流程
- **协作层面**：四位成员分工明确，通过统一接口规范进行模块间联调，培养团队协作能力

## 二、软件结构与设计

### 2.1 系统架构

系统采用**三层架构**，层次清晰，职责分明：

```
┌──────────────────────────────────────────────────────┐
│                  表示层（Qt GUI）                      │
│  LoginDialog │ MainWindow │ 各业务Page                │
│  负责用户交互、数据展示、输入校验                        │
└──────────────────────────────────────────────────────┘
                          │
                          ▼ 通过 SystemIntegrator 门面调用
┌──────────────────────────────────────────────────────┐
│                  业务逻辑层                            │
│  RoomManager(A)  ← 被 BookingManager 调用             │
│  BookingManager(B) ← 被 BillingManager 调用           │
│  BillingManager(C)                                    │
│  Logger / Validator / FileHelper (D)                  │
└──────────────────────────────────────────────────────┘
                          │
                          ▼ 文件读写
┌──────────────────────────────────────────────────────┐
│                  数据持久化层                           │
│  rooms.txt │ customers.txt │ bookings.txt             │
│  bills.txt │ system.log                               │
└──────────────────────────────────────────────────────┘
```

### 2.2 模块划分与职责

#### 成员 A：房间体系（Room Hierarchy）

| 类 | 设计模式 | 职责 |
|---|---------|------|
| `Room`（抽象基类） | 抽象类 + 纯虚函数 | 定义所有房型的公共接口，支持多态容器 `vector<Room*>` |
| `StandardRoom` | 单继承 | 基础房型，含床位数、窗户、WiFi |
| `DeluxeRoom` | 单继承 | 中高端房型，含阳台、浴缸、早餐 |
| `SuiteRoom` | 二次继承 `Room→DeluxeRoom→SuiteRoom` | 顶级房型，继承豪华间全部设施，增加客厅、会议室 |
| `RoomManager` | 统一管理器 | 管理所有房间对象，提供增删查改、排序、统计、文件IO |

**OOP 知识点**：抽象类、纯虚函数、二次继承、虚析构函数、多态容器、`dynamic_cast`、运算符重载（`<`、`==`、`<<`）

#### 成员 B：客户与预订体系（Customer & Booking）

| 类 | 设计模式 | 职责 |
|---|---------|------|
| `Customer` | 基类 | 普通客户，含折扣率、最大预订数等虚函数 |
| `VIPCustomer` | 继承 + 多态覆盖 | VIP客户，按等级享受不同折扣（0.95/0.90/0.85） |
| `Booking` | 数据类 | 封装一次预订的完整生命周期 |
| `BookingException` | 自定义异常（继承 `std::exception`） | 10 种业务异常错误码，含 `what()` 和 `getErrorCode()` |
| `BookingManager` | 统一管理器 | 预订 CRUD、入住/退房流程、日期冲突检测、客户管理、文件IO |

**OOP 知识点**：继承多态、二次继承（Customer→VIPCustomer）、自定义异常类、`throw`/`try`/`catch`、业务流程状态机

#### 成员 C：日期、账单与营收分析（Date & Bill）

| 类 | 设计模式 | 职责 |
|---|---------|------|
| `Date` | 值类型 + 运算符重载 | 封装日期的全部运算：加减天数、比较、输入输出、闰年处理 |
| `Bill` | 数据类 + 友元 | 封装账单，支持生成、折扣、支付、合并运算、类型转换 |
| `BillingManager` | 统一管理器 | 账单 CRUD、营收统计分析（STL 算法重点） |

**OOP 知识点**：运算符重载（`+`、`-`、`++`/`--`、全部比较运算符、`<<`/`>>`）、友元函数/类、`operator double()` 类型转换、STL 算法（`accumulate`、`count_if`、`copy_if`、`sort` + lambda）

#### 成员 D：工具类与系统集成（Utilities & Integration）

| 类 | 设计模式 | 职责 |
|---|---------|------|
| `Logger` | **单例模式** | 全局唯一日志实例，支持 INFO/WARNING/ERROR 级别，`operator()` 重载 |
| `Validator` | 纯静态工具类 | 8 种数据校验方法（房间号、手机号、身份证、日期等） |
| `FileHelper` | 纯静态工具类 | 通用文件读写工具（读行、写行、追加、备份） |
| `SystemIntegrator` | **门面模式** | 统一入口，封装三个 Manager + Logger，对外提供简洁业务接口 |
| Qt GUI 界面 | MVC 模式 | 8 个页面类，通过 SystemIntegrator 交互，不直接操作 Manager |

**OOP 知识点**：单例模式（私有构造 + 静态实例 + 禁拷贝）、门面模式、异常包装（`executeSafe`）、Qt 信号槽机制

### 2.3 模块交互关系

```
Qt UI  ──→  SystemIntegrator（门面）
                │
                ├──→  RoomManager  ──→  Room*（多态）
                ├──→  BookingManager  ──→  Customer*（多态）、Booking、RoomManager*
                ├──→  BillingManager  ──→  Bill、Booking、Room、Customer
                └──→  Logger（单例）
```

**关键设计原则**：
- 表示层**只通过 SystemIntegrator 访问业务逻辑**，不直接操作任何 Manager
- Manager 之间**通过指针引用协作**（如 BookingManager 持有 RoomManager*）
- 所有 Manager 的数据**通过文件进行持久化**，系统启动时加载，关闭时保存

## 三、AI 引导过程及代码人工调整

### 3.1 AI 引导过程

本项目采用 **"AI 辅助设计 + 人工实现 + AI 辅助调试"** 的协作模式，各阶段分工如下：

#### 阶段一：需求分析与架构设计（AI 主导）

- 使用 AI（Claude）进行需求分析，梳理系统的功能模块与类层次关系
- AI 生成详细的《详细设计文档》，明确每位成员的类设计任务、OOP 知识点覆盖、模块间接口规范
- 四位成员根据设计文档讨论并确定分工

#### 阶段二：头文件接口设计（人工 + AI）

- 各成员先自行设计 `.h` 头文件，定义类的成员变量和方法签名
- 使用 AI 审查头文件设计的合理性（如析构函数是否虚函数、const 修饰是否正确）
- 统一接口后各成员独立开发

#### 阶段三：核心逻辑实现（人工为主）

- **成员 A/B/C** 的核心业务逻辑（Room 体系、Booking 流程、Date 运算、BillingManager 统计）由成员人工编写，体现 OOP 设计能力
- 成员在实现过程中遇到问题时使用 AI 辅助排查（如跨月日期加减的边界处理、STL 算法的 lambda 写法）

#### 阶段四：集成与 UI 开发（AI 主导）

- **成员 D** 的工具类（Logger/Validator/FileHelper）和 SystemIntegrator 门面类由 AI 辅助生成框架代码，人工调整细节
- **Qt GUI 界面**完全由 AI 辅助生成，包括 LoginDialog、MainWindow、5 个业务页面、CMakeLists.txt 的 Qt 集成配置
- AI 负责解决 Qt 编译环境配置（brew 安装 Qt、cmake 配置、AUTOMOC 启用）

#### 阶段五：联调与 Bug 修复（人工 + AI）

- 集成各模块后，AI 辅助排查编译错误和运行时异常
- 典型问题：数据文件含注释行导致 `stoi` 崩溃，由 AI 定位并修复 `loadRooms`/`loadCustomers`/`loadBookings` 的注释行过滤

### 3.2 代码人工调整

以下为人工审查后对 AI 生成代码所做的关键调整：

| 调整项 | 说明 |
|-------|------|
| **Bill 类友元声明** | AI 生成的 Bill.h 中新增 `friend class BillingManager`，使 BillingManager 的 `loadBills` 可直接访问私有成员，简化反序列化逻辑 |
| **Bill/BillingManager 完整实现** | 成员 C 将初始的 `// TODO` 空实现补全为完整的方法体，包含 STL 算法（accumulate、count_if、copy_if）和 lambda 表达式 |
| **数据文件注释行过滤** | RoomManager::loadRooms、BookingManager::loadCustomers/Bookings 原始代码未过滤 `#` 开头行，人工补上 `if (line[0] == '#') continue;` |
| **SystemIntegrator 扩展** | 为支持 Qt UI 展示全部数据，新增 `getAllBookings()` 和 `getAllBills()` 接口 |
| **实时自动保存机制** | 新增 `saveAllData()` 私有方法，在每次增删改操作（添加/删除房间、注册/删除客户、创建预订、入住/退房、支付等）成功后立即调用，将所有数据写入文件，防止程序异常退出时数据丢失 |
| **房间与客户信息编辑** | Room 基类新增 `setFloor()`，Customer 基类新增 `setIdCard()`，SystemIntegrator 新增 `updateRoom()` 和 `updateCustomer()` 接口，UI 页面提供编辑对话框 |
| **客户删除功能** | SystemIntegrator 新增 `deleteCustomer()` 接口，删除前校验是否有活跃预订，CustomerPage 新增删除按钮 |
| **Qt 界面样式调整** | 人工调整颜色方案、按钮 hover 效果、表格交替行颜色，提升视觉体验 |
| **登录对话框交互优化** | 增加回车键跳转焦点、密码框回车触发登录、登录失败清空密码等细节 |

## 四、项目演示

### 4.1 系统启动与登录

运行 `./HotelManagement`，弹出登录对话框。默认管理员账号 `admin/admin`，登录成功后进入主界面。

### 4.2 仪表盘

展示系统概览：总房间数、空闲房间数、客户总数、未付账单数，以及完整的系统状态报告。

### 4.3 房间管理

- 点击「标准间/豪华间/套房」按钮弹出表单，填写房间信息后添加
- 选中行后点击「编辑」可修改房间楼层和价格
- 选中行后点击「删除房间」可删除
- 表格实时展示所有房间的房间号、楼层、价格、状态、类型、描述

### 4.4 客户管理

- 注册普通客户或 VIP 客户（选择等级 1-3）
- 选中普通客户可升级为 VIP
- 选中任意客户点击「编辑」可修改姓名、电话、身份证信息
- 选中客户点击「删除」可删除无活跃预订的客户
- 表格展示客户 ID、姓名、电话、类型、累计消费、折扣率

### 4.5 预订管理

- 新建预订：选择客户 ID、房间号、入住/退房日期
- 办理入住：选择待入住预订，点击办理入住
- 办理退房：选择已入住预订，系统自动生成账单并显示金额明细
- 取消预订：对待入住预订进行取消
- 续住：延长已入住预订的退房日期

### 4.6 账单与统计

- 账单列表展示所有账单（已付/未付状态用颜色区分）
- 支付选中账单
- 统计报表展示系统完整状态
- 消费排行 Top 5 客户表格

### 4.7 CLI 测试（补充）

除 Qt GUI 外，项目还保留了 CLI 测试入口 `src/main.cpp` 和 `src/test_main.cpp`，可通过终端直接运行验证核心业务逻辑。

## 五、评分要素对应

### 5.1 项目价值

- **实用性**：系统覆盖酒店运营核心流程（房间→客户→预订→账单→统计），功能闭环完整
- **可扩展性**：三层架构解耦，新增房型/客户类型/统计维度只需扩展对应模块
- **技术完整性**：同时具备 CLI 和 Qt GUI 两种运行方式，数据文件持久化支持重启

### 5.2 设计合理性

- **层次清晰**：表示层（Qt）→ 业务逻辑层（Manager）→ 数据层（文件），职责单一
- **设计模式运用恰当**：
  - **单例模式**（Logger）：全局唯一日志实例，避免文件竞争
  - **门面模式**（SystemIntegrator）：统一入口，降低 UI 层与业务层的耦合
  - **抽象类 + 多态**（Room 体系）：`vector<Room*>` 统一管理不同房型
  - **自定义异常**（BookingException）：10 种业务错误码，异常传播链完整
- **模块间依赖合理**：RoomManager → BookingManager → BillingManager，单向依赖，无循环

### 5.3 复杂度

#### 设计复杂度

| 复杂度维度 | 体现 |
|-----------|------|
| **继承层次** | 三层继承链 `Room → DeluxeRoom → SuiteRoom`；`Customer → VIPCustomer` |
| **多态** | Room 抽象基类 + 3 个具体子类；Customer 基类 + VIPCustomer 覆盖虚函数 |
| **运算符重载** | Date 类 12 个运算符重载；Bill 类 `operator+`、`operator<<`、`operator double()` |
| **STL 运用** | `vector`、`map`、`sort`、`accumulate`、`count_if`、`copy_if`、`find_if`、lambda |
| **异常处理** | 自定义异常类 + try-catch 链 + executeSafe 异常包装 |
| **设计模式** | 单例模式、门面模式、抽象工厂（多态创建）、静态工具类 |
| **GUI 复杂度** | Qt 框架，8 个页面类，信号槽机制，对话框表单，CMake 构建 |
